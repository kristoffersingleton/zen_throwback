CSS Zen Garden — Evolved

Zen Throwback

A portable, dynamic presentation-layer engine. The same content, rendered in radically different ways — chosen by the visitor.

What is this?

In 2003, the CSS Zen Garden demonstrated that a single HTML document could look radically different depending solely on its stylesheet. It was a landmark moment — but the experience was static. You'd load a new URL to see a new theme.

Zen Throwback picks up that thread and pulls it into the present. Themes switch instantly, without navigation. Your preference persists across sessions. Share a link with ?zen=terminal and friends see the same view. And if enough sites adopt the discovery protocol, a browser extension could apply your preferred aesthetic everywhere.

Level 1

Token

Override ~20 CSS custom properties. Instant, zero flash, works on any site that builds against the token contract.

Level 2

Layer

Inject a full stylesheet via CSS @layer. CSS Zen Garden–style total control, without !important hacks.

Level 3

Component

Swap Web Component registrations to change structure, not just style. Requires deferred element registration.

Level 4

Generative

A JS function produces CSS from a seed. Same seed = same theme, anywhere. True random-every-load capability.

Generative themes

The Generative theme produces a unique palette, type stack, and layout from a short seed string. The same seed always produces the same result — so seeds are shareable.

Adoption protocol

Any site can opt into the Zen ecosystem with two lines in <head>:

<!-- 1. Declare the manifest (makes themes discoverable) --> <link rel="zen-manifest" href="/zen-manifest.json"> <!-- 2. Declare token version (signals CSS var contract) --> <meta name="zen:tokens" content="v1">

Then load the engine and optionally drop in the switcher UI:

// ES module usage import Zen from '/src/zen.js'; import '/src/zen-switcher.js'; const zen = new Zen(); await zen.init(); // loads manifest, applies initial theme window.zen = zen; // expose for zen-switcher and console // API zen.apply('terminal'); // switch to a theme zen.random(); // pick a random theme zen.next(); // cycle through themes zen.reset(); // restore default + clear storage // Events zen.addEventListener('zen:change', e => console.log(e.detail.theme)); zen.addEventListener('zen:ready', e => console.log(e.detail.manifest));
<!-- Drop-in UI widget --> <zen-switcher position="bottom-right"></zen-switcher>
The token contract

Themes and sites communicate through a standard set of CSS custom properties. Site authors write CSS against these variables; themes override them. A token-only theme needs no knowledge of your site's markup.

/* In your site's CSS — build against the contract */ .card { background: var(--zen-color-surface); border: var(--zen-border-width) var(--zen-border-style) var(--zen-color-border); border-radius: var(--zen-radius); font-family: var(--zen-font-body); transition: border-color var(--zen-transition); } /* In a token theme — just override the vars */ :root { --zen-color-bg: #0d1117; --zen-color-text: #39ff14; --zen-font-body: ui-monospace, monospace; }
The interesting open question is whether this becomes a W3C proposal or stays a community convention — like rel="canonical" or OpenGraph tags. The barrier to adoption is intentionally low.
Future directions

Some directions this could grow:

  • Browser extension — reads rel="zen-manifest", applies user's global preference to every site that supports it
  • Theme registry — a CDN-hosted index of community themes, shareable by URL
  • CSS Houdini (Paint API) — procedural backgrounds and textures once Firefox/Safari ship it
  • OS-level signal — respond to prefers-color-scheme, prefers-contrast, prefers-reduced-motion as first-class theme parameters
  • Per-origin storage — browser extension persists theme preference per-domain
  • Theme composition — combine a layout theme + a colour theme + a type theme independently
